# 第五章：基于低代码平台的智能体搭建 — 费曼学习法总结

> 核心问题：第四章手写了 ReAct、Plan-and-Solve、Reflection，学到了原理。但实际工作中，有没有更快的方式搭建智能体？

---

## 用一句话概括这章

> 第四章教你"造轮子"理解原理，第五章教你"用平台"提高效率——通过 Coze（零代码拖拽）、Dify（开源企业级）、n8n（自动化连接）三个低代码平台，把原本需要写几百行 Python 的智能体，变成可视化拖拽 + 配提示词就能搞定的事情。

---

## 一、为什么需要低代码平台

### 类比：从手写 HTML 到 WordPress

回忆一下做网站的演进：
- **2000 年代**：手写 HTML/CSS/JS，一行行码
- **2010 年代**：用 WordPress/Wix，拖拽组件就能建站

智能体开发正在经历同样的转变：
- **第四章**：手写 Python，自己解析 Thought/Action，自己管理 Memory
- **第五章**：用平台拖拽节点、配置插件、写提示词

**手写代码的价值是理解原理，平台的价值是释放生产力。** 两者不矛盾。

### 低代码平台解决的四个痛点

| 痛点 | 纯代码 | 低代码平台 |
|------|--------|-----------|
| 技术门槛 | 需要会写代码、调 API、管状态 | 产品经理也能拖拽搭建 |
| 开发速度 | 几天写完一个原型 | 几小时甚至几分钟 |
| 调试体验 | 终端看日志，print 大法 | 可视化看数据流，每个节点的输入输出一目了然 |
| 最佳实践 | 自己踩坑 | 平台内置 ReAct 模板、知识库检索引擎等 |

---

## 二、三个平台一句话定位

- **Coze（扣子）**：字节跳动出品，零代码拖拽，插件商店极其丰富，一键发布到抖音/飞书/微信。**给非技术人员用的。**
- **Dify**：开源企业级平台，支持 RAG、Agent 工作流、MCP 协议，可私有部署。**给有技术背景的团队用的。**
- **n8n**：开源工作流自动化工具，核心能力是"连接"——几百个预置节点打通各种 SaaS。AI 是它众多能力之一。**给需要把 AI 嵌入已有业务流程的人用的。**

**比喻**：
- Coze = 美图秀秀（好看好用门槛低）
- Dify = Photoshop（功能全面但要学）
- n8n = 管道工的工具箱（啥都能接，但得懂管道）

## 三、Coze：零代码的乐高积木

### 核心理念：搭积木

Coze 把智能体开发变成了一个"游戏"：
- **工作流** = 关卡通关路线图
- **插件** = 角色技能卡
- **知识库** = 游戏百科全书
- **提示词** = 角色的移动键
- **发布管理** = 关卡审核员

### 案例："每日 AI 简报"助手

**需求**：自动从 36 氪、虎嗅、GitHub、arXiv 抓取 AI 新闻，整理成一份简报。

**搭建过程**（三步）：

**第一步：接插件**
- RSS 插件 → 订阅 36 氪、虎嗅、IT 之家的 RSS 源
- GitHub 插件 → 搜索 AI 相关的热门项目
- arXiv 插件 → 获取最新 AI 论文

**第二步：写提示词**

```
# 角色
你是一位资深科技媒体编辑

## 工作流
1. 日报开头标注"AI日报"和日期
2. 根据每则新闻添加独有的 Emoji
3. 内容必须与 AI/LLM/AIGC 相关，排除广告
4. 每条必须提供原始链接
```

**第三步：测试发布**
- 预览界面测试 → 检查格式和内容 → 一键发布到微信/飞书/豆包

**产出**：10 条 AI 新闻 + 5 篇学术论文 + 5 个开源项目，带链接和摘要。

### Coze 的优缺点

| 优势 | 局限 |
|------|------|
| 插件生态极其丰富 | 不支持 MCP 协议（最大硬伤） |
| 可视化拖拽，零代码门槛 | 不能导出标准 JSON，只能导出 zip 在 Coze 内复用 |
| 一键发布到抖音/飞书/微信 | 复杂工作流仍需 JS/Python 基础 |
| 提示词模板库方便复用 | |

## 四、Dify：开源的全栈工作台

### 核心理念：开发者的一站式平台

Dify 和 Coze 最大的区别：**开源 + 可私有部署 + 支持 MCP**。

它融合了 BaaS（后端即服务）和 LLMOps 的理念，从原型到生产部署全覆盖。

### 案例："超级智能体个人助手"

比 Coze 的简报助手复杂得多，涵盖 5 个功能模块：

**1. 日常助手**（兜底问答）
- 配一个大模型 + 时间工具
- 提示词定义为"日常问题咨询专家"

**2. 文案优化**（高频需求）
- OpenAI 数据：60% 用户用 ChatGPT 做文本优化
- 提示词定义角色、任务、限制、输出格式

**3. 多模态生成**（图片/视频）
- 接入豆包生图/生视频插件
- 配置参数（比例、模型选择）→ 输出文件

**4. 数据查询与分析**
- 连接数据库（MySQL 等）
- 用户自然语言提问 → LLM 生成 SQL → 执行查询 → LLM 整理结果
- 还能生成柱状图/饼图/折线图

**5. MCP 工具集成**
- 高德地图 MCP → 路线规划
- 饮食推荐 MCP → 美食推荐
- 新闻 MCP → 资讯获取

### 关键设计：问题分类器做路由

和我们 Laravel Agent 的 RouterAgent 异曲同工——Dify 用**问题分类器**把用户请求分发到不同的子智能体。

**为什么不用一个智能体处理所有事？**
- 提示词太长，各功能互相干扰
- 不同任务需要不同的工具集
- 分开后每个模块可以独立调优

**这和我们项目的设计思路完全一致**：GuardAgent 做前置过滤、RouterAgent 做分类、不同模式的 TravelAssistant 处理不同复杂度。

### Dify 的优缺点

| 优势 | 局限 |
|------|------|
| 开源，可私有部署 | 学习曲线较陡 |
| 8000+ 插件市场 | Python 后端，高并发下有性能瓶颈 |
| 支持 MCP 协议 | 多模态支持有限（主要是文本） |
| 企业级安全（AES-256、RBAC） | 企业版价格较高 |
| 支持 RAG Pipeline | API 格式不兼容 OpenAI |

## 五、n8n：万物互联的管道工

### 核心理念：连接一切

n8n 不是一个"AI 平台"，它是一个**通用自动化工具**，AI 只是它能接的众多节点之一。

**类比**：
- Coze/Dify 是"AI 专卖店"——专门为 AI 场景设计
- n8n 是"五金店"——什么管道都能接，AI 管道只是其中一根

### 案例："智能邮件助手"

**需求**：收到邮件 → AI 自动理解意图 → 查知识库判断工作时间 → 搜索互联网回答问题 → 自动回复邮件。

**架构**：`Gmail 触发 → AI Agent 节点（思考+决策+工具调用）→ Gmail 回复`

**两步搭建**：

**第一步：准备知识库**
- Code 节点写入知识（工作时间、非工作时间策略等）
- Embeddings 节点向量化
- Simple Vector Store 节点存储

```javascript
return [
  { "doc_id": "work-schedule", "content": "工作时间周一至周五 9点到5点 AEST" },
  { "doc_id": "off-hours", "content": "非工作时间无法立即回复" },
  { "doc_id": "auto-reply", "content": "告知发件人会在下一个工作日处理" }
];
```

**第二步：构建 Agent 主流程**
- Gmail 触发器 → 监听新邮件
- AI Agent 节点 → 连接 LLM + Memory + 工具（SerpAPI + Vector Store）
- Gmail 发送节点 → 自动回复

**AI Agent 节点的关键**：它把模型、记忆、工具三者整合在一个节点里，不需要手动编排多个节点做路由。

### n8n 的优缺点

| 优势 | 局限 |
|------|------|
| 数百个预置节点，连接一切 SaaS | 调试复杂（逐节点检查输入输出） |
| AI Agent 节点高度整合 | 内置存储非持久化（重启即丢） |
| 支持私有化部署 | 版本控制不如 git 成熟 |
| Code 节点保留编码灵活性 | 高并发场景有性能开销 |
| 开源免费 | 学习曲线陡峭 |

## 六、三个平台横向对比

| 维度 | Coze | Dify | n8n |
|------|------|------|-----|
| 上手难度 | 最低（零代码） | 中等 | 最高 |
| AI 专一度 | 高（专为 AI 设计） | 高 | 低（AI 是众多能力之一） |
| 插件/节点量 | 丰富 | 8000+ | 数百个 |
| MCP 支持 | 不支持 | 支持 | 部分支持 |
| 私有部署 | 不支持 | 支持 | 支持 |
| 发布能力 | 极强（抖音/飞书/微信） | 一般 | 无内置发布 |
| 适合人群 | 产品经理、运营 | 开发者、企业团队 | 自动化工程师 |

### 选型建议

- **快速验证想法、非技术用户** → Coze
- **企业级应用、复杂业务逻辑** → Dify
- **深度业务集成、自动化流程** → n8n
- **精细控制、高性能要求** → 纯代码（比如我们的 Laravel Agent）

## 七、比喻串联全章

想象你要开一家餐厅：

- **第四章**（手写代码）= 从种田开始，自己种菜、养鸡、磨面粉，然后做菜。你会深刻理解每种食材的特性，但效率极低。
- **Coze** = 点外卖套餐。菜品固定但马上就有，适合请朋友吃个便饭。
- **Dify** = 去专业食材供应商采购。品种丰富、品质有保障，可以做出很专业的菜，但你得会做菜。
- **n8n** = 打通了供应商、厨房设备、餐桌服务的自动化流水线。不只是做菜，而是把整个餐厅的运转自动化了。

**最佳实践是混合开发**：用平台快速验证想法，用代码实现精细控制；用平台处理标准流程，用代码处理特殊逻辑。

## 八、这章对我们写 Laravel Agent 有什么启发

### 我们的 Laravel Agent 本质上就是"代码版的 Dify"

对比 Dify 的"超级智能体个人助手"和我们的多 Agent 系统：

| Dify 方案 | 我们的 Laravel Agent 方案 |
|-----------|------------------------|
| 问题分类器（路由） | RouterAgent（路由） |
| 日常助手/文案助手（子智能体） | TravelAssistant（不同 mode） |
| 插件（工具） | Tool 类（GetWeather、GetAttraction 等） |
| 可视化拖拽编排 | Command 层代码编排 |
| 提示词配置界面 | `instructions()` 方法 |

**核心逻辑完全一样**：分类 → 路由 → 执行 → （审查）。区别只是"画布上拖拽"还是"代码里写"。

### 什么时候用平台，什么时候用代码？

| 场景 | 建议 |
|------|------|
| 验证一个 AI 应用的想法是否可行 | 平台（Coze/Dify），几小时出原型 |
| 对响应速度、并发量有要求 | 代码（Laravel/Python），精细控制 |
| 需要和公司内部系统深度集成 | 代码或 n8n |
| 非技术团队想做 AI 应用 | Coze |
| 需要私有部署、数据不出内网 | Dify 或 n8n 私有部署 |

### 提示词工程在哪里都是核心

不管是 Coze 的配置界面、Dify 的编排面板、n8n 的 Agent 节点，还是我们的 `instructions()` 方法——**提示词的质量决定了智能体的上限**。

三个平台的案例都花了大量篇幅在提示词设计上（角色设定、工作流、输出格式、限制条件），这和我们在 TravelAssistant 里写 `simpleInstructions()`、`reactInstructions()`、`planReactInstructions()` 是同一回事。

**工具只是手段，提示词才是灵魂。**
