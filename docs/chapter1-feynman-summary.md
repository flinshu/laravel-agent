# 第一章：初识智能体 — 费曼式知识总结

> 用最简单的语言解释复杂的概念，如果你不能简单地解释它，说明你还没有真正理解它。 — 理查德·费曼

## 一句话说清楚：智能体是什么？

智能体就是一个**能感知周围环境、自己决定下一步做什么、然后动手去做的程序**。

想象你请了一个私人助理：你说"帮我规划一趟北京之旅"，助理会自己查天气、挑景点、订酒店，中间不需要你每一步都指挥。这就是智能体——你给目标，它自己想办法。

## 四个核心要素

| 要素 | 类比 | 在旅行助手中 |
|------|------|-------------|
| **环境** | 助理工作的"世界" | 天气 API、搜索引擎、用户对话 |
| **传感器** | 助理的"眼睛和耳朵" | 接收用户输入、读取 API 返回值 |
| **执行器** | 助理的"手" | 调用天气查询、调用搜索工具 |
| **自主性** | 助理的"大脑" | LLM 决定先查天气再推荐景点 |

关键区分：普通程序是"你说一步我做一步"，智能体是"你说目标我自己规划步骤"。

## 智能体的演进：从恒温器到 ChatGPT

把智能体的发展想象成一个人的成长过程：

```
反射智能体 → 基于模型 → 基于目标 → 基于效用 → 学习型 → LLM 智能体
  婴儿反射    记住事物    有了目标    会权衡利弊    从经验学习    通用大脑
```

1. **反射智能体**：温度高了就开空调。纯条件反射，没有记忆。
2. **基于模型**：隧道里看不到前车了，但"记得"它还在前面。有了记忆。
3. **基于目标**：不只是反应，而是"我要到公司"，然后规划路线。有了规划。
4. **基于效用**：不只是到公司，还要最快、最省油、不堵车。会权衡。
5. **学习型**：AlphaGo 从零开始下棋，越下越强。会进化。
6. **LLM 智能体**：不用手写规则，大模型"懂"自然语言，能泛化到各种任务。

**核心跃迁**：传统智能体的能力来自人写的规则；LLM 智能体的能力来自海量数据中学到的隐式知识。

## 三种分类维度

### 维度一：决策架构（它内部多复杂？）

反应式 → 模型式 → 目标式 → 效用式 → 学习式（上面已讲）

### 维度二：时间特性（它想多久才行动？）

- **反应式**：毫秒级响应。安全气囊、高频交易。快但短视。
- **规划式**：深思熟虑。棋手思考十步。准但慢。
- **混合式**（最实用）：底层快速反应 + 上层长远规划。LLM 智能体天然就是混合式——每一轮"Thought→Action→Observation"就是一次"想一步做一步"的微循环。

### 维度三：知识表示（它用什么方式"思考"？）

- **符号主义**：像图书管理员，用逻辑规则推理。透明但脆弱。
- **亚符号主义**：像小孩看了一万张猫的图片就认识猫。强大但黑箱。
- **神经符号混合**（= 卡尼曼的双系统理论）：
  - 系统 1（快思考）= 神经网络的直觉识别
  - 系统 2（慢思考）= 符号化的逻辑推理
  - **LLM 智能体恰好融合了两者**：神经网络做感知 + 结构化输出做推理

## 核心运行机制：Agent Loop

整个智能体的工作就是一个不断重复的循环：

```
         ┌────────────────────────────────┐
         │                                │
         ▼                                │
  ① 感知(Perception)                      │
     接收用户指令 / 上一步的结果              │
         │                                │
         ▼                                │
  ② 思考(Thought)                         │
     ├─ 规划：当前该做什么？                  │
     └─ 工具选择：用哪个工具？什么参数？        │
         │                                │
         ▼                                │
  ③ 行动(Action)                          │
     调用选定的工具                          │
         │                                │
         ▼                                │
  ④ 观察(Observation)                     │
     工具返回结果，反馈给感知 ───────────────┘
```

**用旅行助手的例子走一遍**：

| 轮次 | Thought | Action | Observation |
|------|---------|--------|-------------|
| 1 | 用户要查北京天气，先调天气工具 | `get_weather("北京")` | 晴天，26°C |
| 2 | 晴天适合户外，调景点推荐工具 | `get_attraction("北京", "Sunny")` | 推荐颐和园、长城 |
| 3 | 信息够了，给出最终答案 | `Finish[推荐颐和园...]` | — |

每一轮就是一次"感知→思考→行动→观察"的闭环。3 轮完成任务。

## Thought-Action-Observation 协议

这是让 LLM 和外部工具协作的"接口协议"：

- **Thought**：LLM 输出的思考过程（自然语言）
- **Action**：LLM 决定调用的工具（结构化的函数调用）
- **Observation**：工具执行后返回的结果（自然语言化的反馈）

解析器（Parser）负责从 LLM 的文本输出中提取 Action，调用对应工具，再把结果格式化为 Observation 喂回去。

## PEAS 模型：定义任务环境

设计智能体前先用 PEAS 描述它的"工作环境"：

| 字母 | 含义 | 旅行助手的例子 |
|------|------|-------------|
| **P** | Performance 性能度量 | 推荐准确性、用户满意度 |
| **E** | Environment 环境 | 天气 API、搜索引擎、用户对话 |
| **A** | Actuators 执行器 | 调用 API、生成文本回复 |
| **S** | Sensors 传感器 | 用户输入、API 响应数据 |

环境特性：部分可观察、随机性、多智能体、序贯、动态。

## 两种协作模式

### 模式一：智能体作为工具（辅助人类）

代表产品：GitHub Copilot、Claude Code、Cursor。人类主导，AI 辅助完成代码、测试等重复工作。

### 模式二：智能体作为自主协作者

代表框架：CrewAI、MetaGPT、LangGraph、AutoGen。给一个高层目标，智能体自己规划、执行、反思，像团队成员一样工作。

## Workflow vs Agent

| | Workflow | Agent |
|---|---------|-------|
| 本质 | 预设的流程图 | 自主的目标追求者 |
| 决策 | if-else 规则 | LLM 动态推理 |
| 灵活性 | 只能处理预设场景 | 能应对新场景 |
| 适用场景 | 规则明确、需要可预测性 | 规则模糊、需要灵活性 |

**一句话区分**：Workflow 是"你按步骤执行"，Agent 是"你自己想办法达成目标"。
