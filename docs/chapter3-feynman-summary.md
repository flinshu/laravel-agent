# 第三章：大语言模型基础 — 费曼学习法总结

> 核心问题：GPT 这样的大模型，到底是怎么工作的？它的"智能"从何而来？

---

## 用一句话概括这章

> LLM 的本质是一个**超级文字接龙机器**：先通过概率统计学会预测下一个词，再用注意力机制看懂全局上下文，最后靠海量数据训练出"世界知识"——这三件事合在一起，就产生了我们今天看到的"智能"。

---

## 一、从统计到神经网络：语言模型的进化

### 第一代：N-gram — 统计词的出现频率

**核心思想**：一个词出现的概率，只和它前面几个词有关。

```
Bigram（N=2）：P("learns" | "agent") = 出现"agent learns"的次数 / 出现"agent"的次数
```

**计算例子**：

语料库：`datawhale agent learns` + `datawhale agent works`

```
P(agent | datawhale) = 2/2 = 1.0     ← datawhale 后面总是 agent
P(learns | agent)    = 1/2 = 0.5     ← agent 后面一半是 learns
P(整句)              ≈ 0.33 × 1.0 × 0.5 = 0.167
```

**致命缺陷**：

| 问题 | 表现 |
|------|------|
| 数据稀疏 | 没见过的词组，概率直接为 0 |
| 不理解语义 | `agent` 和 `robot` 语义相近，但对模型来说完全不同 |

---

### 第二代：词嵌入 — 把词变成向量

**核心思想**：把每个词映射到高维空间里的一个点，语义相近的词，点的位置也相近。

```
agent  → [0.9, 0.8, ...]
robot  → [0.85, 0.79, ...]   ← 和 agent 距离很近
apple  → [0.1, 0.3, ...]     ← 和 agent 距离很远
```

**最著名的例子**：

```
King - Man + Woman ≈ Queen
```

用向量做减法和加法，就能做"语义平移"。这证明词向量不只是坐标，而是编码了性别、皇室这类抽象概念。

**用余弦相似度衡量"距离"**：

```
similarity(a, b) = cos(θ) = (a · b) / (|a| × |b|)

结果为 1  → 完全相同方向（高度相关）
结果为 0  → 垂直（毫无关系）
结果为 -1 → 相反方向（负相关）
```

---

### 第三代：RNN — 给模型加"记忆"

**N-gram 的问题**：上下文窗口固定，只能看前 N 个词，再早的信息全丢。

**RNN 的方案**：引入隐藏状态（hidden state），把上一步的记忆传给下一步：

```
h(t) = f( 当前词向量 + 上一步的隐藏状态 h(t-1) )
```

**问题**：序列太长时，梯度在反向传播中经过多次连乘，要么消失（接近 0），要么爆炸（极大）——导致模型学不到序列早期信息的影响。

**LSTM 的解法**：引入**细胞状态**（独立的信息通路）+ **门控机制**：

- 遗忘门：决定丢掉什么旧信息
- 输入门：决定记住什么新信息
- 输出门：决定输出什么给下一步

> **比喻**：RNN 像短暂的工作记忆，读完一篇文章，开头的内容已经忘了。LSTM 像记笔记的人，主动选择哪些值得记，哪些可以划掉。

## 二、Transformer：彻底抛弃循环，全靠注意力

### 为什么抛弃 RNN？

RNN 必须按顺序处理，第 t 步必须等第 t-1 步完成——**无法并行**。序列越长越慢，限制了模型规模。

### 自注意力机制：每个词同时看所有词

**直觉**：读 "The **agent** learns because **it** is intelligent" 时，理解 `it` 需要往前找到 `agent`。自注意力让每个词都能动态决定该关注哪些词。

**实现方式**：为每个词生成三个角色：

| 角色 | 含义 |
|------|------|
| Q（Query）| 当前词在"问"：谁跟我最相关？ |
| K（Key）  | 每个词的"标签"：我是什么样的词？ |
| V（Value）| 每个词的"内容"：我携带什么信息？ |

**计算公式**：

```
Attention(Q, K, V) = softmax( QKᵀ / √dk ) × V
```

- `QKᵀ`：点积，计算"问题"和所有"标签"的匹配程度
- `/ √dk`：除以维度的平方根，防止数值太大梯度消失
- `softmax`：转成概率分布（所有权重加和为 1）
- `× V`：按权重对内容加权求和

**多头注意力**：把一次注意力拆成 h 组并行计算，每组从不同角度看：

```
头1：关注指代关系（it → agent）
头2：关注时态关系（learns → present）
头3：关注从属关系（intelligent → agent）
...
最后拼接所有头的结果
```

### 位置编码：告诉模型词的顺序

自注意力本身不知道词序——"agent learns" 和 "learns agent" 对它一样。

解决方案：给每个位置加一个**位置向量**（用正弦/余弦函数生成），让相同词在不同位置有不同的输入表示。

### 编码器-解码器整体结构

```
输入句子
    ↓
编码器（Encoder）：理解整个输入，为每个词生成富含上下文的向量
    ↓
解码器（Decoder）：逐词生成输出，每步参考编码器的理解结果
    ↓
输出句子
```

## 三、Decoder-Only：GPT 的简洁哲学

### 核心思想

OpenAI 的 GPT 问了一个问题：**语言的本质，不就是预测下一个词吗？**

无论回答问题、写代码还是讲故事，本质都是：给定前文，生成下一个最合理的词。

于是 GPT **直接扔掉编码器，只保留解码器**。

### 自回归生成（Autoregressive）

```
输入：Datawhale Agent is
↓ 预测下一个词 → "a"
输入：Datawhale Agent is a
↓ 预测下一个词 → "powerful"
输入：Datawhale Agent is a powerful
↓ ...不断重复
```

就是文字接龙，一个词一个词地生成。

### 掩码自注意力：不能"偷看"未来

在训练时，需要防止模型用第 t+1 个词来预测第 t 个词（作弊）。

解决方案：在注意力计算中加掩码，把当前位置之后的分数设为负无穷 → softmax 后变为 0 → 数学上阻止了向后看。

### 为什么 Decoder-Only 更成功？

| 优势 | 原因 |
|------|------|
| 训练目标统一 | 只做"预测下一个词"，无标注数据就能训练 |
| 结构简单 | 更容易扩大规模（GPT-4 千亿参数都是这个架构）|
| 天然适合生成 | 对话、写作、代码生成都是生成任务 |

## 四、与 LLM 交互：提示工程

### 采样参数控制输出风格

**Temperature（温度）**：控制随机性。

```
T → 0：每次都选最高概率的词，输出确定、保守、重复
T → 大：概率分布变平，随机性增加，输出多样、创意、有时不连贯
```

场景对应：
- 代码生成、数学计算 → 低温度（< 0.3）
- 日常对话、文案写作 → 中温度（0.3 ~ 0.7）
- 诗歌创作、头脑风暴 → 高温度（0.7 ~ 2）

**Top-k**：只从概率最高的 k 个词中采样。k=1 = 贪心采样（完全确定）。

**Top-p（核采样）**：累积概率达到 p 就截止，比 Top-k 更灵活地适应分布变化。

**优先级**：温度调整 → Top-k → Top-p，三者叠加使用。

---

### 提示类型

| 类型 | 示例 | 适用场景 |
|------|------|---------|
| Zero-shot | "这段话是正面还是负面情感？" | 简单明确任务 |
| One-shot | 给一个示例 + 问题 | 需要明确格式 |
| Few-shot | 给多个示例 + 问题 | 复杂格式/细微差别 |

**思维链（Chain-of-Thought）**：加一句"请逐步思考"，让模型先输出推理过程再给答案，显著提升复杂推理准确率。

```
直接问：80场赢60%+15场赢12场，总胜率？ → 容易出错

加上"请一步一步计算"：
第一步：80 × 60% = 48
第二步：总场数 = 95，总胜 = 60
第三步：60/95 ≈ 63.2%  → 准确
```

## 五、分词：文本变数字的过程

### 为什么需要分词

模型只能处理数字，文本必须先转成 Token ID。

| 分词方式 | 问题 |
|---------|------|
| 按词切 | 词表太大；未见过的新词（OOV）处理不了 |
| 按字符切 | 单字符无语义，模型学习效率低 |
| 子词切分 | 折中：常见词保留完整，罕见词拆成有意义的片段 |

### BPE（字节对编码）

GPT 系列使用的算法，核心是**贪心合并**：

```
初始词表：所有字符
↓
统计所有相邻词元对的频率
↓
合并频率最高的那对，加入词表
↓
重复，直到词表大小达到预设值
```

例子：`hug pug pun bun` 语料库

```
第1次：'u' + 'g' → 'ug'（出现频率最高）
第2次：'ug' + '</w>' → 'ug</w>'
第3次：'u' + 'n' → 'un'
第4次：'un' + '</w>' → 'un</w>'
```

新词 "bug" → 词表里没有 "bug"，但有 'b' 和 'ug' → 切成 `['b', 'ug']`。

### 分词对开发者的意义

- **上下文窗口**是按 Token 数计算的，不是字符数。中文 Token 效率比英文低（同样意思需要更多 Token）。
- **API 成本**按 Token 计费，理解分词才能估算成本。
- **奇怪的表现**：`2+2` 和 `2 + 2` 可能被分成不同 Token，导致模型对同一计算的结果不一样。

## 六、缩放法则与局限性

### 缩放法则（Scaling Laws）

模型性能与三个因素成**幂律关系**（对数-对数坐标下是直线）：

```
参数量 ↑ → 性能 ↑
训练数据量 ↑ → 性能 ↑
计算量 ↑ → 性能 ↑
```

**Chinchilla 定律（DeepMind，2022）**：不是越大越好，而是**参数量和数据量要匹配**。

```
700亿参数 + 4倍数据 > 1750亿参数（GPT-3）+ 少量数据
```

数据效率比规模更重要。

**涌现能力（Emergence）**：规模超过某个阈值后，突然出现从未训练过的新能力：
- 思维链推理
- 指令遵循
- 代码生成
- 少样本学习

---

### 模型幻觉（Hallucination）

LLM 的根本缺陷之一：**自信地"编造"不存在的事实**。

| 类型 | 例子 |
|------|------|
| 事实性幻觉 | 引用不存在的论文、编造历史事件 |
| 忠实性幻觉 | 文档摘要和原文意思不一致 |
| 内在幻觉 | 生成内容与输入信息直接矛盾 |

**原因**：自回归机制只是在预测"下一个最可能的词"，没有内置的事实核查模块。

**缓解方案**：

| 方法 | 原理 |
|------|------|
| RAG（检索增强生成）| 生成前先从外部知识库检索相关信息，用事实锚定输出 |
| 工具调用 | 让模型调用搜索引擎、计算器获取精确结果 |
| 思维链 + 验证 | 引导模型分步推理，每步可检查 |

## 八、一个比喻串起全章

把 LLM 的工作比作**一个超级编辑在接龙故事**：

**N-gram 时代**：只记得前面两三个字，靠"这两个字后面通常跟什么"来续写。记忆短，没语义。

**词嵌入**：给每个词在"意义地图"上找到位置，语义相近的词住得很近。现在能"理解"词的含义了。

**RNN/LSTM**：编辑有了笔记本，可以在读到后面时翻查前面记录的内容。但笔记本有限，太长的故事早期内容还是会忘。

**Transformer/自注意力**：编辑桌上摆着整个故事的每一页，写任何一句话时，都能同时扫一眼所有页，决定哪些内容最相关。而且多个编辑并行工作（多头注意力）。

**Decoder-Only（GPT）**：去掉了"先读全文再写"的步骤，直接做接龙——读了多少就续写多少，但每次续写前能看到所有已写内容，且不能偷看后面。

**预训练**：这个编辑在接龙之前，已经把互联网上所有文章都读了一遍，把世界知识都"存"在了脑子里。

**幻觉**：编辑太自信，有时会"顺滑地编造"一个听起来很合理但实际不存在的情节。

**RAG**：在编辑旁边放一个实时检索助理，每次续写前先查一下事实，再动笔。

## 九、这章对我们写 Laravel Agent 的启发

### 1. Temperature 控制推荐风格

我们的旅行助手应该用**中低温度**（推荐具体景点需要确定性），而不是高温度（避免随机推荐奇怪地点）。

可以在 Agent 或 Provider 配置中设置：
```php
#[Model('glm-5.1')]
// temperature 应该在 0.3-0.6 之间，兼顾多样性和准确性
```

### 2. 思维链（CoT）和我们的系统提示

我们的提示词里有"每次只执行一个工具调用"——这其实是在引导 LLM **一步步思考**，而不是一次性输出。这是 CoT 在 Agent 场景的变体。

### 3. 幻觉 → 为什么我们需要工具调用

LLM 对"故宫今天是否有余票"一无所知（知识截止日期问题），且可能幻觉一个看起来合理的答案。

`CheckTicketAvailability` 工具的价值就在这里：**用实时 Tavily 搜索结果锚定 LLM 的输出，替代可能幻觉的回答**。

### 4. Token 窗口 → 对话历史管理

`RemembersConversations` 存储的对话历史会消耗 Token。随着对话越来越长，成本增加、可能超过上下文窗口。

这是 Agent 记忆管理的核心矛盾：记得多 vs 花费少。后续章节会讲如何解决。

### 5. 模型选型 → 为什么我们用 GLM

GLM（智谱）是国产开源模型，支持中文效果好，通过 OpenAI 兼容接口接入，成本低于 GPT-4。对中文旅行推荐场景而言，是合理的成本/性能权衡。
